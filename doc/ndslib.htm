Background Data

Background data varies depending on type of background.

Text Background:

            Appear to work the same as GBA text backgrounds.  16 bit tile indexes for the map data, 16 or 256 color tiles.  The only difference seems to be that bit 4 and 5 are now character base bits allowing selection of base blocks up to base block 16 giving 256 KB of accessible memory.  Map base block still works the same. Text backgrounds can also access the extended palettes.

 

Rotation Background:

            Also appear to work exactly the same as GBA.  8 bit indexes and 256 color only.

 

Extended Rotation Backgrounds:

These have the same 16 bit index found in text backgrounds but can only be 256 color.  They palette index can access the extended palettes for a total of 256x16 colors on screen.

           

If the 256 color bit is set in the control register the background will act as a 256 color bitmap with the following available sizes:

128x128

256x256

512x256

512x512

            If the 256 color bit and bit 2 are set the background becomes a 16 bit background (yes 16 bit meaning high bit is the alpha bit).  The sizes remain the same.

 

Big Bitmap (mode 6 main only)

            This is a large linear addressed bitmap of size 1024x512 or 512x1024. 

 

Frame Buffer (main only)

            Access the frame buffer directly to display 16 bit video. 

 

Extended palettes

            When bit 30 of Display control is set on either display, Text and rotation backgrounds will access an external palette.  These palettes can be stored in VRAM_E, F, G.  The palette used is determined by the palette index in the map tile attribute (high 4 bits) similar to the way it would reference a 16x16 color palette.  There are 16 256-color palettes addressable by each background.

 

Sprite Data

Sprite data appears exactly the same as the GBA differing only in the fact that there are two copies of the OAM and palette.

Texture Formats

The DS supports many texture formats.  More to come on this.

Number Formats

Vertex (v16)

            1.3.12 fixed point number (1 bit sign, 3 bits number, 12 bits fraction)

UV (t16)

            1.11.4 fixed point number.

Normal (v10)

1.0.9 fixed point (generally 3 of them will be packed into a single 32bit value) used for normals and lighting vectors.

Matrix (f32)

            1.19.12 fixed point (matricies have a much larger range of available values)

 

FIFO

            The fifo can be accessed by command followed by data, or you can simply write the data to the address of the fifo + command * 4.  Currently ndslib only uses the latter method.  Playlists can be DMA’d to the FIFO but no support is currently supplied in NDSLIB for playlists.

Touch Pad

Functions

 

DS has a nice touch screen to play with.  This can only be accessed via the arm 7.  The values received from the touch pad controller are not calibrated.  To calibrate you need two known points on the screen.  These can either be taken from the BIOS or they can be manually determined.  See the touch pad demo to see how to use these values to convert actual touch position to the screen position.

Sound

Registers

Functions

 

16 Direct sound channels are available. 

Here is some sample code to play a wave file recorded as 8 bit mono 22.05Khz in windows sound recorder on channel ‘c’.  This is not a great example as the header is played but you get the gist.

This code must be run on the arm7 processor…it will not work on the arm9.

 

  powerON(POWER_SOUND);

 

  SOUND_CR = SCHANNEL_ENABLE | SOUND_VOL(0x3F) ;

 

  SCHANNEL_TIMER(c) = SOUND_FREQ(22050);

  SCHANNEL_SOURCE(c) = (u32)GETRAW(test_wav);

  SCHANNEL_LENGTH(c) = GETRAWSIZE(test_wav);

  SCHANNEL_CR(c) = SOUND_ENABLE | SOUND_VOL(0x3F);

 

c can be any value from 0 to 15.  Max volume is 0x7F.  Playback frequency is determined by the timer.  When it overflows more data is loaded.  It is 16 bit and if you write to it that value determines the starting point of the timer.  To calculate timer frequency use the following:

 

#define SOUND_FREQ(n)   (0x10000 - (0x1000000 / (n)))

 

0x10000 is the max timer value plus 1.

0x1000000 is the timer count rate (16.78Mhz)

 

Power must be enabled then the channel set up.

VramSetMainBanks sets the first 4 banks as specified.  These are the most common banks used in both 2D and 3D.

These functions can be used to set each bank individualy.

Used to restore banks to a prior state.  Sometimes (in the case of texture memory especialy) you can only write to vram when it is set to a certain mode.

Console Print Functions

void consoleInit(u16* font, u16* charBase, u16 numCharacters, u8 charStart, u16* map, u8 pal, u8 bitDepth);

font: A 16 color font.  Must map onto the ascii set but does not have to be complete nor start at 0.

 

charBase: The location in video memory that the font will be loaded into.

 

numCharacters: The number of characters in the font

 

charStart:  The ascii value of the first character (would be 0 if you have a full set)

 

map: The background map that will be rendered into.  You will have to set up the background your self.

 

pal: The 16 color palette you want the font to use.  If you specify a palette greater than 15 it will automatically convert the font to one bit per pixel with each pixel rendered with palette entry 255.

 

bitDepth: This is the bitdepth of the background you are using.  The font will be expanded to 256 color tiles if you specify 256 as the bitdepth.

 

void consoleInitDefault(u16* map, u16* charBase, u8 bitDepth);

 

This will init the console with a default font.  Arguments are same as above.

 

 

void consolePutString(int x, int y, char* s);

void consolePutInt(int x, int y, int d);

void consolePutX(int x, int y, int d);

void consolePutChar(int x, int y, char c);

void consolePutBin(int x, int y, int b);

 

This is used to put values at a specific location on screen.

String: prints a 0 terminated string.

Int: Prints a signed integer.

X: Prints a hexidecimal number.

Char: Prints a character to the screen.

Bin: Prints a binary value to the screen.

Prints a formatted string.  Should support everything you are used to in printf other than floats.  Tabs are set to 3 spaces.

%b  will print a binary value.

Moves the cursor to the specified location.

Math Functions

static inline f32 divf32(f32 num, f32 den)

static inline f32 mulf32(f32 a, f32 b)

static inline f32 sqrtf32(f32 a)

 

Fixed point math functions.  All values are 1.19.12 fixed point numbers.  Hardware is used for divide and sqrt and long data types are used internally to prevent loss of data. These functions do not return until the operation is complete.

 

static inline int32 div32(int32 num, int32 den)

static inline int32 div64(int64 num, int32 den)

static inline int32 sqrt32(int a)

 

Normal integer versions of the math functions.  Hardware divide and sqrt are utilized.  These calls do not return until the result is valid.

 

static inline void crossf32(f32 *a, f32 *b, f32 *result)

static inline f32 dotf32(f32 *a, f32 *b)

static inline void normalizef32(f32* a)

Fixed point trig functions.  These operate on fixed point vectors.  They use hardware math were applicable and take arrays of 1.19.12 of size 3 as arguments.

 

GL Functions

void glLoadMatrix4x4(m4x4 * m);

void glLoadMatrix4x3(m4x3 * m);

void glMultMatrix4x4(m4x4 * m);

void glMultMatrix4x3(m4x3 * m);

void glMultMatrix3x3(m3x3 * m);

 

Load functions can be used to load matrixes into the stack.

Mult functions multiply the current stack matrix by the supplied matrix with the result replacing the stack.   

 

void glRotateXi(int angle);

void glRotateYi(int angle);

void glRotateZi(int angle);

void glRotateX(float angle);

void glRotateY(float angle);

void glRotateZ(float angle);

These are rotation commands.  They multiply the current stack matrix by the applicable rotation (this is a fifo command not an actual matrix operation) and result becomes current stack matrix.  This works in model view and should function in texture mode as well.

 

Functions post fixed with ‘i’ take an integer angle between 0 and 511 which correspond to an angle between 0-359.  The floating point versions simply convert the supplied angle (0-359 degrees) and call the integer version and are supplied for prototyping purposes only.

 

void gluLookAtf32(f32 eyex, f32 eyey, f32 eyez, f32 lookAtx, f32 lookAty, f32 lookAtz, f32 upx, f32 upy, f32 upz);

 

void gluLookAt(float eyex, float eyey, float eyez, float lookAtx, float lookAty, float lookAtz, float upx, float upy, float upz);

 

void gluFrustumf32(f32 left, f32 right, f32 bottom, f32 top, f32 near, f32 far);

void gluPerspectivef32(int fovy, f32 aspect, f32 zNear, f32 zFar);

void gluPerspective(float fovy, float aspect, float zNear, float zFar);

 

These functions operate similar to the glu versions available in openGL.  The floating point version convert all values to integer or fixed point and call the fixed point versions.  The floating point versions are supplied for prototyping purposes only.

 

int glTexImage2D(int target, int empty1, int type, int sizeX, int sizeY, int empty2, int param, uint8* texture);

 

void glBindTexture(int target, int name);

int glGenTextures(int n, int *names);

void glResetTextures(void);

 

Texture functions operate similar to those available in openGL.  Currently there is no support for dynamic allocation of textures, meaning once texture memory is full it must be reset before new textures are loaded.

 

GlGenTextures allocates texture memory and returns a texture ID (or –1 if no more texture memory is available).  It should handle textures that straddle vram banks.

 

GlTexImage2D differs from the gl call as it allows texture parameters to be passed, as well as including the texture format (type) in the argument.

 

void glResetMatrixStack(void);

void glReset(void);

glReset should be called at the beginning of the frame. 

glResetMatrixStack is called internally.

This group of functions are all declared inline.

glPolyFmt has no GL equivalent but it is required to set certain rendering attributes. Culling, alpha blending, lights applied to the poly.

 

DMA Functions

static inline void dmaCopy(void * source, void * dest, uint32 size)

static inline void dmaCopyWords(uint8 channel, void* src, void* dest, uint32 size)

static inline void dmaCopyHalfWords(uint8 channel, void* src, void* dest, uint32 size)

 

Size is in bytes.

Halfword dma addresses must be aligned to 2 byte boundries.

Word dma must be aligned to 4 bytes.

DmaCopy uses channel 3.

 

These DMA calls do not return until the DMA is complete.

 

static inline void dmaCopyAsynch(void * source, void * dest, uint32 size)

static inline void dmaCopyWordsAsynch (uint8 channel, void* src, void* dest, uint32 size)

static inline void dmaCopyHalfWordsAsynch (uint8 channel, void* src, void* dest, uint32 size)

inline int dmaBusy(uint8 channel)

 

Size is in bytes.

Halfword dma addresses must be aligned to 2 byte boundries.

Word dma must be aligned to 4 bytes.

DmaCopy uses channel 3.

 

These DMA calls return once the transfer is started.  No attempt is made to ensure the channel is clear prior to starting the transfer.  A call to dmaBusy should be made between successive DMA calls to the same channel.

//   bit(6..0) when compressed: run length - 3, followed by one byte of true data, to be repeated

#define f32toint(n)          ((n) >> 12)
