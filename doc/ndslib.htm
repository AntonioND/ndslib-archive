Description

            VRAM consists of nine banks of memory that can be mapped to different parts of the GPU.  The banks are as depicted below with each bank having different sizes and purposes.  When banks are used for the 2D core the memory gets mapped to lower VRAM, which starts at 0x0600:0000.  Separate defines are provided for low vram memory access.

 

Vram control registers are 8 bit with the low 3 bits being the current owner of the VRAM, bits 3 and 4 being the offset when mapped or the slot when used as texture or palette memory.  Bit 7 is the enable bit.  Not all of these have been rigorously tested. If you find these to be in error please send a correction (preferably with a demo+source demonstrating this) to dovoto@thepernproject.com .

Background Data

Background data varies depending on type of background.

Text Background:

            Appear to work the same as GBA text backgrounds.  16 bit tile indexes for the map data, 16 or 256 color tiles.  The only difference seems to be that bit 4 and 5 are now character base bits allowing selection of base blocks up to base block 16 giving 256 KB of accessible memory.  Map base block still works the same. Text backgrounds can also access the extended palettes.

 

Rotation Background:

            Also appear to work exactly the same as GBA.  8 bit indexes and 256 color only.

 

Extended Rotation Backgrounds:

These have the same 16 bit index found in text backgrounds but can only be 256 color.  They palette index can access the extended palettes for a total of 256x16 colors on screen.

           

If the 256 color bit is set in the control register the background will act as a 256 color bitmap with the following available sizes:

128x128

256x256

512x256

512x512

            If the 256 color bit and bit 2 are set the background becomes a 16 bit background (yes 16 bit meaning high bit is the alpha bit).  The sizes remain the same.

 

Big Bitmap (mode 6 main only)

            This is a large linear addressed bitmap of size 1024x512 or 512x1024. 

 

Frame Buffer (main only)

            Access the frame buffer directly to display 16 bit video. 

 

Extended palettes

            When bit 30 of Display control is set on either display, Text and rotation backgrounds will access an external palette.  These palettes can be stored in VRAM_E, F, G.  The palette used is determined by the palette index in the map tile attribute (high 4 bits) similar to the way it would reference a 16x16 color palette.  There are 16 256-color palettes addressable by each background.

Character offset works the same as on GBA only now the distance between characters is not fixed at 32 bytes but expands with expanded VRAM to allow access to the entire range of memory with the same 10 bits of character name.

 

Since only one main bank can be assigned to the sub display the max size is 128KB when using extended memory on the sub display.

 

Enabling Bitmap Mode:

            Setting bits 10 and 11 both to one in ATTRIBUTE 0 sets the mode to bitmap mode.  Bitmap objects are 16 bit color objects. 

 

Extended Palettes:

            If bit 31 of DISP_CR is set to 1 then the palette bits in attribute 2 now select from one of 16 256-color palettes stored in one of the specified VRAM banks.  This only applies to tile mapped sprites.

 

Using individual alpha:

            When in bitmap mode the palette indexes in Attribute 2 will be combined with the alpha blend control register settings to determine final alpha blending on the sprite.

Texture Formats

The DS supports many texture formats.  

Only textures currently supported by NDSLIB are 16 bit BGR, and 256 color paletted textures.

 

            16 bit:

                        Same format as 15-bit BGR with the high bit as alpha.  Be sure your textures conversion utilities take this into account.

            8 bit:

                        To use 8 bit palettes NDSLIB expects the palette to immediately follow the pixel data.  You can also manually load the palette into the appropriate VRAM bank.

 

If GL_RGB is specified as a texture format to the texture manager the high bit of each pixel will be set when copied into the texture bank and the internal texture mode will be set to GL_RGBA.

Number Formats

Vertex (v16)

            1.3.12 fixed point number (1 bit sign, 3 bits number, 12 bits fraction)

UV (t16)

            1.11.4 fixed point number.

Normal (v10)

1.0.9 fixed point (generally 3 of them will be packed into a single 32bit value) used for normals and lighting vectors.

Matrix (f32)

            1.19.12 fixed point (matricies have a much larger range of available values)

 

FIFO

            The fifo can either be accessed via display lists copied directly to it or by sending the data to the fifo plus an offset that represents the command.  In other words if you wish to send vertex data to the fifo simply write to (FIFO + (VERTEX_COMMAND * 4)).  Most of the GL port in NDSLIB writes in this manner and has registers defined for each command.

 

            To send display list commands they must first be packed.  Each write to the fifo represents four commands which must then be followed by the data needed for those commands.  The first command goes into the low byte followed by the next command in the next byte.  A command of zero will be interpreted as a NOP with no data required.  NDSLIB expects the first 32 bit value in the list to be the number of words in the list.  Here is a sample display list which would render a triangle.  No support is provided in NDSLIB to generate display lists at this time.

//display list

u32 triangle[] =

{

      12,

      FIFO_COMMAND_PACK(FIFO_BEGIN, FIFO_COLOR, FIFO_VERTEX16, FIFO_COLOR),

      GL_TRIANGLE,

      RGB15(31,0,0),

      VERTEX_PACK(intov16(-1),intov16(-1)), VERTEX_PACK(0,0),

      RGB15(0,31,0),

      FIFO_COMMAND_PACK(FIFO_VERTEX16, FIFO_COLOR, FIFO_VERTEX16, FIFO_END),

      VERTEX_PACK(intov16(1),intov16(-1)), VERTEX_PACK(0,0),

      RGB15(0,0,31),

      VERTEX_PACK(intov16(0),intov16(1)), VERTEX_PACK(0,0),

};

Sound

Registers

Functions

 

16 Direct sound channels are available. 

Here is some sample code to play a wave file recorded as 8 bit mono 22.05Khz in windows sound recorder on channel ‘c’.  This is not a great example as the header is played but you get the gist.

This code must be run on the arm7 processor…it will not work on the arm9.

 

  powerON(POWER_SOUND);

 

  SOUND_CR = SCHANNEL_ENABLE | SOUND_VOL(0x3F) ;

 

  SCHANNEL_TIMER(c) = SOUND_FREQ(22050);

  SCHANNEL_SOURCE(c) = (u32)GETRAW(test_wav);

  SCHANNEL_LENGTH(c) = GETRAWSIZE(test_wav);

  SCHANNEL_CR(c) = SOUND_ENABLE | SOUND_VOL(0x3F);

 

c can be any value from 0 to 15.  Max volume is 0x7F.  Playback frequency is determined by the timer.  When it overflows more data is loaded.  It is 16 bit and if you write to it that value determines the starting point of the timer.  To calculate timer frequency use the following:

 

#define SOUND_FREQ(n)   (0x10000 - (0x1000000 / (n)))

 

0x10000 is the max timer value plus 1.

0x1000000 is the timer count rate (16.78Mhz)

 

Power must be enabled then the channel set up.
